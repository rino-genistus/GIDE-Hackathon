# Incident Post-Mortem: Payment Processing Outage
**Incident ID:** INC-2026-0147  
**Severity:** P1 — Critical  
**Date of Incident:** February 18, 2026  
**Duration:** 2 hours 34 minutes (09:12 – 11:46 UTC)  
**Author:** Site Reliability Engineering Team  
**Status:** Final — Approved  
**Last Updated:** February 24, 2026

---

## Executive Summary

On February 18, 2026, the Apex Commerce payment processing system experienced a complete outage lasting 2 hours and 34 minutes, affecting 100% of payment transactions during that window. Approximately 47,000 transactions failed. Estimated gross merchandise value impacted: $2.3 million. No data was lost. All failed transactions were retried successfully after service restoration.

Root cause: A database connection pool exhaustion triggered by an untested code path in the new fraud detection service deployed the previous day. The connection pool was not sized for the traffic pattern generated by the fraud service's synchronous database calls under load.

## Root Cause Analysis

### Immediate Cause
The fraud-detection-service v1.4.2 introduced a new feature: real-time velocity checks against a historical transactions table in the payments PostgreSQL database. This check ran synchronously on every payment request before the payment was processed.

The velocity check query was not optimized — it performed a full sequential scan on the transactions table (1.4 billion rows) because the relevant composite index (`customer_id, created_at`) had not been added in the migration. Under normal traffic (100–300 requests/second), this caused each payment request to hold a database connection for an average of 340ms rather than the usual 8ms.

The connection pool for the payments database has a maximum size of 150 connections. Under the morning traffic ramp (Apex's peak traffic is 08:00–11:00 UTC), the pool exhausted within approximately 22 minutes of reaching peak throughput.

### Contributing Factors

**1. Missing database index**
The migration script for v1.4.2 created the `transaction_velocity` query function but did not include the composite index required for efficient query execution. The code review did not catch this because the query appeared fast in the development environment (which had only 50,000 rows in the transactions table).

**2. Load testing gap**
The fraud detection service was load tested in the staging environment, but staging uses a transactions table seeded with 500,000 rows — less than 0.04% of production data volume. The performance difference between staging and production was not flagged.

**3. Off-hours deployment without traffic validation**
v1.4.2 was deployed at 23:40 UTC when traffic was approximately 8% of peak. No anomalies were detected at that traffic level because the connection pool was only at 12% utilization. The service appeared healthy for 9 hours before peak traffic exposed the issue.

**4. No connection pool monitoring alert**
While connection pool utilization is tracked in Prometheus, there was no alert configured for high utilization. The payment success rate alert fired, but if connection pool alerts had been configured, the issue may have been caught 15–20 minutes earlier.

**5. Synchronous database call in payment critical path**
Architectural: the fraud check was implemented as a synchronous blocking call in the payment processing flow. This created a direct coupling between fraud service database performance and payment availability. The design should have used an asynchronous pattern or a dedicated read replica.

## What Went Well

- On-call response was fast — alert acknowledged in 3 minutes, P1 declared within 10 minutes.
- Rollback decision was made quickly (within 13 minutes of alert) without excessive deliberation.
- ArgoCD rollback executed cleanly with no complications — our GitOps process worked as designed.
- All failed transactions were retried automatically with no data loss — the idempotency design of our payments system held.
- Communication to stakeholders was clear and timely — the CEO and VP of Customer Success were notified within 20 minutes.

## What Went Poorly

- The missing index was not caught in code review or staging testing.
- Off-hours deployment created a false sense of security — the service "passed" its deployment window without real load.
- No connection pool monitoring meant we were flying blind on a critical resource.
- The fraud check architectural decision (synchronous, in critical path) was a latent risk that should have been flagged in design review.

---

## Lessons Learned

The most important lesson from this incident is not the missing index — that's a fixable process gap. The deeper lesson is that **production data volume is a first-class performance variable** and our testing infrastructure did not reflect it. Any query that performs well with 500,000 rows can fail catastrophically with 1.4 billion rows. Fixing our staging data volume is the highest-priority structural action item.

The architectural lesson is equally important: external service dependencies should never be placed synchronously in the critical path of a payment transaction. A fraud check that is slow or unavailable should degrade gracefully (fall back to async review), not bring down payments entirely.
